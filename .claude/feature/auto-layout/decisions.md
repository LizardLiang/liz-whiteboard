# Decisions Log — Auto Layout

## Product Decisions (Athena — PRD Creation)

- **Algorithm: force-directed via `d3-force`.** Rationale: produces organic clusters that match how users intuitively read FK structure (related things close, unrelated things apart); `d3-force` is already a project dependency (no new install) and the user explicitly confirmed this choice during clarification. Rejected: hierarchical/ELK layout (`elkjs` is also installed) — better for strictly tree-structured data flows but produces rigid grid-like results that don't reflect _clusters_ of mutually-related tables; defer as a possible future "alternative layout" mode. Rejected: tree layout — assumes a root, which ER schemas with cyclic relationships do not have.

- **Trigger: manual button only.** Rationale: opt-in is safe for existing users; a continuous / live re-layout would surprise users mid-edit and risks fighting their drags. Rejected: auto-trigger on table-add — too disruptive while editing. Rejected: re-layout on import — no import flow exists yet to hook into.

- **Scope: full diagram, all tables.** Rationale: simplest mental model ("one button, lays out everything"), matches the user's stated success criteria; selection-based partial layout introduces tricky questions about boundary handling between laid-out and fixed regions. Rejected: selection-based layout — explicitly out of scope for v1; can be added later without breaking the v1 contract.

- **Existing positions: overwritten unconditionally.** Rationale: user explicitly said "clean slate, overwrite all"; preserving partial positions is a worse mental model and produces hybrid layouts that look chaotic. Rejected: "freeze pinned tables" mode — adds UX surface area (pin/unpin) without v1 demand. Rejected: "merge with current layout" — ill-defined; force-directed simulations don't naturally support partial constraints without per-node weights, which is more UX than v1 needs.

- **Success criteria are testable, not subjective.** Rationale: avoids "this layout looks bad" review bikeshedding. Each P0 success metric (no overlap, FK-pairs closer than non-FK pairs, viewport coverage) maps to an automated test. Rejected: a purely visual "looks good" QA gate — subjective and would block CI/CD.

- **Performance target: ≤ 2s for 100 tables on a mid-range laptop, no UI freeze.** Rationale: 100 tables matches the user's stated upper bound ("up to ~100 tables without freezing"); 2s is the threshold above which users typically wonder if the system is broken. Rejected: hard 500ms target — likely unachievable for 100-table force simulations without tradeoffs in layout quality. Rejected: no performance bound — risks shipping a feature that freezes the browser on real schemas.

- **Persistence and collaboration reuse existing channels.** Rationale: positions produced by Auto Layout are _just positions_ — semantically identical to a manual drag — and should travel through the existing save and Socket.IO sync paths. This avoids inventing a new "layout-applied" event and keeps the server contract unchanged. Rejected: a dedicated `LayoutApplied` event — would force every collaborator to know about Auto Layout as a concept, when in fact they only need the new positions.

- **No animation requirement in v1.** Rationale: snap-to-final-positions is sufficient to satisfy every P0 acceptance criterion; animation is polish. Listed as P2 (FR-030) so it can ship later without re-PRD'ing. Rejected: animation as P0 — adds risk and time without changing the core value.

- **No tunable parameters in the UI.** Rationale: forces, link distance, iteration count are _implementation_ concerns; surfacing them in the UI bloats the toolbar and demands user knowledge of force-directed layouts. Rejected: an "advanced" panel — defer until users actually ask for it.

- **Failure leaves the diagram in pre-click state.** Rationale: a partial layout (some tables moved, some didn't) is the worst outcome — overlap, lost mental model, hard to recover. All-or-nothing is safer. Rejected: "best-effort partial application" — would violate FR-005 atomicity.

- **No PRD-level decision on update batching.** Rationale: whether to send N individual position updates or one bulk update is a tech-spec / performance concern, not a product concern. Flagged in Risks for Hephaestus to resolve. Rejected: prescribing "must use bulk update" in the PRD — overreach into tech decisions.

- **No new external API or service.** Rationale: every dependency needed is already in `package.json` (`d3-force`, `@xyflow/react`). Rejected: server-side layout computation — adds a round-trip, makes the feature offline-incapable, and gains nothing for a 100-table workload that runs comfortably in the browser.

## Revision Requests

### Nemesis (2-prd-review, 2026-05-01) — REVISIONS verdict, 8 BLOCKING items

1. FR-003 ambiguous on "FK-related" definition + edge cases (empty graph, fully-connected, isolated tables, circular FKs).
2. FR-004 ambiguous overlap contract ("0 overlap or 16px gap" — pick one).
3. FR-007 "no input lag > 200ms" undefined measurement; reconcile with disabled-button re-entry rule.
4. Performance target unbound — "mid-range laptop" not reproducible.
5. FR-009 multi-user sync contradiction between FR-005 atomicity and streamed per-node updates.
6. Persistence-failure UX delegated to "existing behaviour" — undefined.
7. Appendix A row 2 (bulk persistence) treated as assumption; Nemesis demanded validation.
8. No cancellation mechanism documented for long-running layouts.

## Athena Revision Resolutions (revision 1.1, 2026-05-01)

- **FR-003 — "FK-related" definition.** Decision: a pair (A, B) is FK-related iff a `Relationship` row links them (equivalently a React Flow edge), direction-agnostic. **Direct edges only — transitive relationships do NOT count.** Rationale: the success metric (60% proximity ratio) needs a binary partition that maps cleanly to a unit test; "same connected component" would make a fully-connected schema have zero non-FK pairs and break the denominator. Fallback specified for empty-FK-set (skip metric), fully-connected (skip metric), isolated tables (count in non-FK side only), circular references (each direct pair counts once). Rejected: 2-hop or transitive-closure-based definitions — harder to test, no clear product benefit.
- **FR-004 — single overlap contract.** Decision: **≥ 16px L∞ gap** between every pair of axis-aligned bounding boxes. Rationale: a single testable contract is required; 16px gives visible breathing room and lines up with d3-force's `forceCollide` configurability. Rejected: 0px overlap (visually too tight, 1px overlap would technically violate but be invisible). Rejected: pixel-distance from centre (depends on table size).
- **FR-007 — main-thread responsiveness measurement.** Decision: violation is any longtask entry of `≥ 200ms` reported by `PerformanceObserver({ entryTypes: ['longtask'] })` during the run. Rationale: this is the W3C-standard measurement; cheap to instrument; integrates with CI. Reconciled with re-entry policy: button is disabled synchronously, so "click responsiveness" applies to the _rest_ of the UI (canvas, sidebar, other toolbar buttons), not to re-clicking Auto Layout (which is intentionally a no-op while disabled). Rejected: input-event-to-handler timing — harder to instrument and noisy.
- **Performance target — reference benchmark hardware.** Decision: bind to a concrete spec (4-core / 2.5GHz, 8GB RAM, Chrome latest, no throttle, AC power) with p95 over 5 runs as the metric, plus a CI proxy (4 vCPU / 8GB Linux runner). Rationale: "mid-range laptop" was untestable; a reproducible benchmark is what Cassandra needs. Rejected: pinning to a specific CPU model — too narrow and will age badly.
- **FR-009 — multi-user sync contract.** Decision: option (b) — streamed per-node updates via the new `table:move:bulk` Socket.IO event (one broadcast carrying all positions, fired by the server only after a successful `updateTablePositionsBulk` transaction). Rationale: option (a) "atomic broadcast that pauses all clients" would require a new client-side state machine (suspend rendering during a remote layout) and is out of scope; option (b) reuses the proven last-write-wins per-node pattern with one new event type. Tradeoffs accepted explicitly: collaborator-side flicker (single-tick on bulk receipt rather than per-node), mid-drag local-wins until release, near-simultaneous double-click resolves by last-write-wins. Rejected: pure per-table `table:move` events (would produce 100 broadcasts and visible piecewise rearrangement on the collaborator's screen — Nemesis's specific complaint). Rejected: a new "freeze remote canvas during layout" UX pattern — material new client state.
- **Persistence-failure UX.** Decision: error toast with Retry action, optimistic local positions kept (never snap back), explicit local/remote divergence acknowledgement in the toast text. Rationale: matches modern optimistic-UI conventions; gives the user a clear path forward without losing their work. Rejected: snap back to pre-layout positions on failure — destroys the user's just-applied work and feels punishing. Rejected: silent retry forever — hides failures from the user.
- **Bulk persistence — validated and added as P0 requirement.** Validated: read `src/lib/server-functions.ts:113-141` and `src/hooks/use-whiteboard-collaboration.ts:207-217`. Confirmed no client-callable bulk path exists (`computeAutoLayout` at line 204 is a _server-side_ layout that bulks DB writes via `prisma.$transaction` but does not emit Socket.IO and is not the path Auto Layout will use because we compute client-side). New requirement: `updateTablePositionsBulk` server function + `table:move:bulk` Socket.IO event. Rationale: Nemesis correctly observed that 100 sequential `updateTablePosition` calls would blow the 2s budget; a single transaction is the only safe path. Rejected: throttling/debouncing per-table calls — still 100 server round-trips, just spread over time, doesn't fix correctness.
- **FR-011 — no in-flight cancellation; pre-run warning for > 50 tables.** Decision: do not implement cancel/Esc in v1; require a confirmation dialog when N > 50. Rationale: safe cancellation requires either a Web Worker with abort signals or cooperative tick-level cancellation with rollback — both are material v1 scope expansion for a feature that targets ≤ 100 tables in ≤ 2s. The pre-run dialog gives users informed consent for the rare > 50 table case where they may notice the wait. Rejected: implementing Esc-to-cancel — rollback semantics are non-trivial (does cancel mean revert to pre-click? what about partial post-pass nudge state? what about the bulk save being mid-flight?). Rejected: silent no-cancel with no warning — would surprise users on > 100 table diagrams.

### Nemesis (2-prd-review round 2, 2026-05-01) — REVISIONS verdict (post-v1.1)

1. **BLOCKING — FR-009 ↔ NFR Persistence ↔ NFR Collaboration ↔ Risks Row 3 ↔ Error Flows internal contradiction.** v1.1 chose option (b) (streamed per-node `table:move`) in FR-009 prose, but every other section (NFR Persistence, NFR Collaboration, Error Flows, Risks Row 3) describes option (a) — server-emitted `table:move:bulk` applied in one tick. The two are mutually exclusive observable behaviours; an engineer cannot satisfy both. Recommended fix: commit to option (a) throughout (it's what NFR Persistence already requires).
2. **MAJOR — FR-011 dialog accessibility.** The new pre-run confirmation dialog has no a11y contract: focus-trap, initial focus target, Esc behaviour, screen-reader role, and announcement on open are all unspecified. For a P0 dialog gating a destructive action this needs explicit ACs.
3. **MINOR — FR-004 "Equivalent test" wording.** The parenthetical claims `gapSum ≥ 16` is "false only when boxes overlap" — mathematically wrong (also false when boxes are gapped < 16px). The contract intent is clear from prose, but the math should be fixed before tech spec.

## Athena Revision Resolutions (revision 1.2, 2026-05-01)

- **FR-009 — REVERSED to option (a) atomic bulk-broadcast.** v1.1's choice of option (b) was internally inconsistent with the rest of the PRD, which already committed to the new bulk persistence path; Nemesis's catch is correct. Decision: rewrite FR-009 as option (a) — client computes positions, emits a single `table:move:bulk` via the new `updateTablePositionsBulk` server function; server validates, transacts in one `prisma.$transaction`, broadcasts a single server-side `table:move:bulk`; collaborators apply in one render tick. FR-005 atomicity is now preserved on **both** local and remote screens. Mid-drag conflict resolution unchanged (local drag wins until release; on drag-end the existing per-table `table:move` overwrites). Concurrent Auto Layout clicks resolved by last-write-wins per whiteboard at the server (whichever transaction commits second is final). All references in NFR Persistence, NFR Collaboration, NFR Reliability, Risks Row 3, Error Flows, and User Flow steps 8-9 are now consistent. Rejected: keeping option (b) and weakening NFR Persistence — that would force the new bulk persistence path to be torn out, regressing the round-1 BLOCKING #7 fix. Rejected: a hybrid (per-node sync + bulk persistence) — produces piecewise rearrangement on collaborator screens, the exact concern Nemesis raised against streamed sync.
- **FR-011 dialog accessibility.** Decision: explicit P0 ACs added — `role="alertdialog"`, focus-trap while open, **initial focus on Run Layout** (the primary action; the user already invoked Auto Layout, so the primary path receives focus and Esc gives them the safe out), Esc dismisses (= Cancel), AT announcement via `aria-labelledby` (title) + `aria-describedby` (description), focus returns to toolbar button on close. Rationale: Nemesis is correct that for a P0 destructive-action gate the a11y contract must be explicit; ambiguity here would either ship an inaccessible dialog or force the tech spec to invent the contract from scratch. Rejected: initial focus on Cancel — would feel "stuck" for keyboard users who want to confirm; the primary action is the user's expressed intent, the dialog is only a second-guess.
- **FR-004 — math cleanup.** Decision: replaced the confused parenthetical with a single correct formula: `L∞ gap = max(A.left - B.right, B.left - A.right, A.top - B.bottom, B.top - A.bottom)` asserted ≥ 16 for every pair (A, B). Rationale: the L∞ gap is exactly the contract intent — negative when overlapping, zero when touching, positive equal to per-axis separation when apart — and the formula is unambiguously testable. The 16px contract itself is unchanged; only the explanatory math is fixed. Rejected: Nemesis's recommended "max(...) ≥ 16" formulation — adopted exactly as recommended.
